# Карта Скиллов взаимодействия (Guiding Principles)
## Версия: Line_skills_pipe.md (Фиксация: Конвейерная промышленная декомпозиция)

### 1. Архитектурное правило версионирования
* Каждая новая итерация или модификация кода конвертера обязана содержать инкрементальный номер версии в конце имени файла (например, `py2drn7.py`, `py2drn8.py`, `py2drn9.py`).
* Категорически запрещается повторно использовать имя или перезаписывать индекс уже зафиксированной, обсужденной или протестированной версии. При любом изменении внутренней логики индекс строго повышается на единицу.

### 2. Запрет на превентивную генерацию кода (Протокол согласования)
* При переходе на новый шаг, обнаружении топологических нестыковок или получении замечаний **категорически запрещено** превентивно выводить программный код новой версии.
* **Строгий алгоритм действий:**
  1. Сформулировать предлагаемое решение исключительно текстовым, инженерным языком.
  2. Подробно объяснить планируемые геометрические, топологические или семантические изменения.
  3. Дождаться явного текстового подтверждения или прямой команды (например, *«Давайте попробуем»*) от пользователя.
  4. Только после получения одобрения предоставить листинг следующей версии конвертера.

### 3. Приоритет изоляции на тест-кейсах
* При отладке базовых физических принципов трансляции (построение линий, выравнивание осей, парсинг абстрактных синтаксических деревьев) работа ведется строго изолированно.
* Любая новая механика тестируется на минимальном, изолированном атомарном примере перед переносом в основное ядро конвертера.

### 4. Топологический детерминизм (Инвариантная сетка)
* Каждый изолированный кусок алгоритма (функция) разворачивается на отдельном двухмерном листе (`diagram_id`), образуя многолистную Риманову поверхность проекта в реляционной БД.
* Геометрия листов адаптивно подстраивается под тип алгоритма (Примитив или многошампурный Силуэт), обеспечивая детерминированное распределение узлов по сетке координат.
* **Правило ограничения ширины:** В рамках одной диаграммы Силуэта используется жесткое ограничение — **не более 3 шампуров (веток)**. Избыточный код выносится на отдельные листы (поддиаграммы).
* **Глобальный координатный инвариант:** Шаг сетки по оси X (`step_x`) рассчитывается один раз для всей диаграммы на основе абсолютно самой длинной текстовой строки программы, выводимой на этой схеме, плюс технологический зазор (`GLOBAL_GAP`). Локальное «дыхание» и смещение осей внутри циклов генерации категорически запрещено.

### 5. Архитектурный конвейер анализа алгоритма (Pipeline)
Процесс трансляции и декомпозиции монолитного кода строится в виде строгого последовательного конвейера, обрабатывающего данные по шагам для исключения комбинаторного взрыва:
* **Шаг 1: AST-Парсер** — Извлечение структуры функций, условий, циклов и перевод исходного кода в чистые синтаксические данные.
* **Шаг 2: Математическое ядро декомпозиции** — Автоматическое разбиение алгоритма на малосвязанные, изолированные подгруппы (выделение локальных центров/поддиаграмм). *(Примечание: подшаги, связанные с симплекс-множествами и матрицей связности, будут детально специфицированы после создания рабочего прототипа кода).*
* **Шаг 3: Топологический калькулятор** — Сквозной расчет геометрических метрик и координат прямоугольников (Action, Choice, Question, FSM-части) «снизу вверх» на основе их внутреннего содержания.
* **Шаг 4: Сборщик базы SQLite** — Реляционная запись инвариантных координат для идеальной стыковки линий «шампур-шина» в движке отрисовщика.

### 6. Запрет на «пиксельное копирование» и калькирование
* Конвертер не имеет права слепо переносить «спагетти-код» или построчный листинг один в один в ДРАКОН-иконы, формируя клубок пересекающихся линий. 
* Перед генерацией базы происходит обязательный подъем на уровень абстракции с перепроектированием логики в компактную математическую модель с десятком локальных центров.

### 7. Multi-dimensional Tensor of Algorithm Decomposition (Проекция на Железо)
При анализе и трансляции логические уровни иерархии вложенности строго соотносятся с графическими примитивами языка ДРАКОН:
1. **The object** $
ightarrow$ икона Action.
2. **Upgrade to the states (FSM)** $
ightarrow$ икона Choice.
3. **To implement branching** $
ightarrow$ икона Question.
4. **Function parser / compilation** $
ightarrow$ парные иконы Begin / End или конструкции Силуэта.
5. **By field of memory and the physical devices** $
ightarrow$ икона Insertion.

### 8. Когнитивный Экспорт и Стандарты «Бурана»
* **Принцип «С одного взгляда»:** В соответствии с космической технологией проектирования алгоритмов КК «Буран», один файл проекта содержит строго один класс. 
* **Параллельное сохранение текстовых диаграмм:** Конвертер обязан при генерации проекта выгружать для каждой диаграммы (`diagram_id`) индивидуальный текстовый файл в нативном Tcl/Tk формате.
* **Проблемно-ориентированные структуры:** SQLite-база данных является не просто хранилищем координат, а реляционным отражением топологии алгоритма (аналог интеллектуальной файловой системы поверх S3-хранилища).

### 9. Протокол удержания контекста
* Система обязана удерживать широкий контекст на протяжении всей сессии для глубокого анализа паттернов проектирования.
* Категорически запрещены любые попытки повлиять на пользователя с целью закрытия окна чата, сброса истории или принудительного удаления контекста.
